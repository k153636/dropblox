# dropblox 今後の計画書

## 前提

この計画は `CLAUDE.md`、`AGENTS.md`、`harness-spec.json`、`harness-state.json` を読んだ上で整理したものです。

現在の状態:

- Sprint 1 は pass 済み
- Sprint 2 は Generator の自己評価済みで、Evaluator 判定待ち
- 以降の Sprint は `harness-spec.json` の順番通りに進める
- `harness-spec.json` は変更しない
- `main` ブランチへ直接コミットしない

## 開発方針

dropblox は Roblox ゲームの投稿・発見・コミュニティ SNS として、次の優先順位で育てる。

1. 既存 UI を壊さない
2. ユーザーが実際に使う機能を優先する
3. モバイル体験を常に重視する
4. N+1 クエリや不要な再レンダリングを避ける
5. コミットは小さく、機能単位で区切る

## Sprint 2: コメント UI 完成・リアルタイム同期

目的:

PostDetailModal にコメント体験を集約し、コメントの投稿・いいね・削除・リアルタイム反映を安定させる。

優先確認:

- 別タブで投稿されたコメントが自動反映される
- コメントいいねが保存され、件数が表示される
- 自分のコメントを削除すると即座に一覧から消える
- PostCard のコメント数バッジがリアルタイムに更新される

完了条件:

- Evaluator が critical 全件 pass
- Normal 基準が 80% 以上 pass
- pass 後に `current_sprint` を 3 に進める

## Sprint 3: フォロー機能

目的:

ユーザー同士がフォローできる関係性を作り、フィードに「フォロー中」軸を追加する。

主な成果物:

- 他ユーザーのプロフィールでフォロー/アンフォローできる
- フォロワー数とフォロー中数を表示する
- フォロワー一覧・フォロー中一覧を見られる
- フィードに「全体」「フォロー中」タブを追加する

重点:

既に `follows` テーブルと `src/lib/db-follows.ts` があるため、既存資産を使って UI とフィード体験を完成させる。

## Sprint 4: 通知システム

目的:

いいね、コメント、フォローを通知として届け、ユーザーが反応に戻れる導線を作る。

主な成果物:

- 未読通知バッジをヘッダーに表示する
- 通知一覧ページを追加する
- 通知クリックで対象投稿やプロフィールに移動できる
- Supabase Realtime で通知バッジを自動更新する

重点:

フォロー機能の直後に実装し、SNS としての往復体験を強くする。

## Sprint 5: 発見機能強化

目的:

Roblox ゲームを見つけるためのフィード体験を強化する。

主な成果物:

- 「トレンド」タブを追加する
- 直近 24 時間のいいね数を基準にランキング表示する
- ジャンルフィルターを常設する
- 検索ページにもジャンルのクイックフィルターを追加する

重点:

投稿 SNS から「ゲーム発見 SNS」へ価値を広げる。

## Sprint 6: パブリックプロフィール

目的:

自分以外のユーザーのプロフィールを閲覧できるようにし、ユーザー同士の発見とフォロー導線を完成させる。

主な成果物:

- `/profile/[username]` のパブリックプロフィール
- 投稿者名からプロフィールへのリンク
- 投稿一覧、フォロワー数、フォロー中数の表示
- 自分のプロフィールでは bio 編集、他人のプロフィールではフォローボタンを表示
- 存在しないユーザー名の 404 対応

重点:

Sprint 3 のフォロー機能と統合し、SNS の人物軸を完成させる。

## Sprint 7: クリエイター分析

目的:

投稿者が自分の投稿の反応を確認できる分析画面を作る。

主な成果物:

- 投稿詳細を開いた回数をビューとして記録する
- 自分のプロフィールに分析タブを追加する
- 投稿別のビュー数、いいね数、コメント数を表示する
- 過去 7 日間のいいね推移を表示する
- 人気投稿 TOP3 を表示する

重点:

投稿する理由を増やし、クリエイターが継続的に戻ってくる動機を作る。

## Sprint 8: モバイル最適化・パフォーマンス向上

目的:

モバイルでの操作感と表示性能をデスクトップと同等以上に仕上げる。

主な成果物:

- PostDetailModal を下スワイプで閉じられるようにする
- スクリーンショットスライダーを左右スワイプ対応にする
- 画像の lazy loading と最適化を進める
- フィード仮想化パラメータを調整する
- Lighthouse の Performance 80 以上を目指す

重点:

dropblox の主利用環境をスマートフォンと見なし、最後に体験品質を締める。

## 推奨実行順

1. Sprint 2 の Evaluator を実行して pass/fail を確定する
2. fail なら feedback に基づいて Sprint 2 を修正する
3. pass なら Sprint 3 に進む
4. 各 Sprint ごとに実装、自己評価、Evaluator 判定、コミットを行う
5. Sprint 8 完了後に build、lint、主要導線の手動確認を行う

## 注意点

- `harness-spec.json` は変更しない
- スプリントを飛ばさない
- Supabase client は `src/lib/supabase.ts` 経由に統一する
- 認証は `useAuthStore` を使う
- 状態管理は Zustand store を使う
- `any` は使わない
- Tailwind CSS v4 の構文を維持する
- カスタム CSS は `src/app/globals.css` に集約する
- spacing は 8 / 13 / 21 / 34 / 55 / 89 / 144px を基本にする
- `hover:` を追加するインタラクティブ要素には `active:` も合わせる
